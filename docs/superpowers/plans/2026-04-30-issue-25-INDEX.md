## Recommended ship order

1. **Plan 1 first.** It's the smallest change with the largest user-pain-reduction — it directly resolves the repro in the issue.
2. **Plan 2 next.** It's independent of Plan 1 and tackles the contributing factor (drifted worktrees missing `.grove/`). Can be implemented in parallel with Plan 1 if a teammate is available.
3. **Plan 3 last, and only if needed.** Validate after Plans 1 + 2 ship: rerun the original issue's repro and check whether the broader stack-health UX is still confusing. If yes, do Plan 3. If no, close issue #25 and open a small ticket for the `non_blocking_services` config knob if/when needed.

## Shared assumptions

These hold across all three plans:

- The user's external-mode setup uses `[plugins.docker.external]` with at least `path`, `env_var`, and `services` configured.
- `docker compose` is in PATH (already enforced by `grove doctor`).
- The shared compose stack uses `--env-file` semantics that grove already supports via `ExternalComposeConfig.EnvFile`.
- TOML parsing rejects unknown keys, so adding a new field to `TestConfig` or `ExternalComposeConfig` requires touching the struct (Tasks 1 in Plans 1 and 3).

## Validation checkpoints

After each plan ships, validate against the **original repro** in [issue #25](https://github.com/lost-in-the/grove/issues/25):

1. **After Plan 1:** `grove test <worktree>` should succeed even when a one-shot init service in the shared stack is failing. The user should see no aborted-test error.
2. **After Plan 2:** in a worktree created via `git worktree add` (drifted), running any grove command should print a one-line warning suggesting `grove adopt`. `grove doctor` should report the drift in Tier 2.
3. **After Plan 3:** `grove up` should succeed in the presence of a failing non-blocking service. `grove ps` should show "up" rather than "degraded" when only a non-blocking service is unhealthy.

## What NOT to do across these plans

- **Don't merge `non_blocking_services` (Plan 3) into `services` semantics.** They're separate lists for a reason: `services` is "what to manage," `non_blocking_services` is "which of those may legitimately exit." Conflating them removes a degree of expressiveness.
- **Don't make `grove adopt` (Plan 2) skip hooks.** The whole point is that drifted worktrees missed the post-create hook fire — adopt is the only path that re-runs them.
- **Don't make `--no-deps` (Plan 1) configurable per-worktree.** It's a project-wide policy. If a worktree's tests need different deps semantics, that's a different problem (probably a dedicated test command).
